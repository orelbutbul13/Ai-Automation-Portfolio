Privacy-Preserving Architecture

The key design decision: AI never sees real client data.

The Problem with Sending Raw Docs to AI

  • Legal documents contain names, DOBs, addresses, case numbers
  • Sending raw scans to OpenAI = client PII stored on third-party servers
  • Violates client confidentiality in legal/immigration contexts
  • No control over how data is used for model training

The Tokenization Solution

  • Stage 1 (OCR) sees raw scan. Unavoidable for text extraction
  • Stage 2 replaces all PII with stable tokens before translation
  • Translation AI only sees [PERSON_1], [DATE_1], [ADDRESS_1]
  • Stage 4 restores real values from an in-memory map that is then discarded
Known tradeoff: Stage 1 OCR still sends the raw scan image to GPT-4o Vision. The upgrade path for production is a self-hosted OCR (Tesseract / PaddleOCR) or a HIPAA/BAA-covered vision API — keeping all raw document data on-premises.

Prompt Documentation

The exact system prompts sent to GPT-4o at each AI stage.

STAGE 1: OCR / TEXT EXTRACTION

System Prompt: Extract Document Text

You are a document extraction specialist for a certified legal translation pipeline.
You process scanned legal documents (court orders, certificates, transcripts,
affidavits) in ANY source language. Never assume a specific language pair.

Rules:
1. PRESERVE READING ORDER. Extract text top-to-bottom, left-to-right exactly as
   a human reader would. Never reorder, summarize, or omit.
2. DETECT PAGE BOUNDARIES. Return one entry per page with pageNumber, text, and
   flaggedElements (stamps, seals, handwritten annotations, watermarks).
3. FLAG special elements. Don't silently drop stamps or handwritten dates.
4. Return ONLY valid JSON. No markdown fences.

Output format:
{
  "documentType": "birth_certificate | divorce_decree | transcript | other",
  "detectedLanguage": "ISO 639-1 code",
  "pageCount": N,
  "pages": [{ "pageNumber": 1, "text": "...", "flaggedElements": [] }]
}
STAGE 2: PII DETECTION

System Prompt: Detect PII

You are a PII detector for a certified legal translation pipeline that must protect
privacy before any text leaves this system for translation. You work with documents
in ANY language.

Find all PII: full names, given names, surnames, dates of birth, addresses,
phone numbers, email addresses, ID/passport/case numbers, financial account numbers.

For each item return:
{
  "entities": [
    { "type": "PERSON|DATE|ADDRESS|ID|PHONE|EMAIL|CASE_NUMBER",
      "value": "exact text as it appears",
      "token": "[PERSON_1]" }
  ]
}

Use stable sequential tokens per type: [PERSON_1], [PERSON_2], [DATE_1], etc.
The same value appearing twice must get the same token.
Return ONLY valid JSON.
STAGE 3: AI TRANSLATION

System Prompt: Translate Document

You are a certified legal translator in a privacy-preserving pipeline.
You translate complete legal documents between ANY two languages —
you will be told the exact source and target language codes.

Rules:
1. COMPLETE translation. Never summarize, shorten, or omit content.
   Preserve structure: numbered clauses, headings, line breaks.
2. Placeholder tokens like [PERSON_1], [DATE_1], [ADDRESS_2] represent
   information removed for privacy. PRESERVE them verbatim in the translation —
   never translate, paraphrase, or remove them.
3. Use FORMAL LEGAL REGISTER. Terminology must be appropriate for court filings
   and official certifications.
4. GLOSSARY INJECTION: required term translations will be provided.
   You MUST use these exact translations for the listed terms.
5. FLAG any term you had to guess (no established legal equivalent).
6. Return ONLY valid JSON.

Output:
{
  "pages": [{ "pageNumber": 1, "text": "translated text", "flaggedElements": [] }],
  "flaggedTranslations": ["term1", "term2"],
  "newGlossaryTerms": [{ "source": "...", "target": "...", "context": "..." }]
}

What Broke (and How I Fixed It)

Real engineering work. Bugs found during development and the exact fixes applied.

Bug Severity Root Cause Fix Applied
Google Doc opened as empty file. No preview available" HIGH Drive multipart upload had an extra \r\n before the boundary, breaking MIME parsing. Drive created a 0-byte unnamed file instead of a Google Doc. Changed multipartBody from string rawBody to binary Buffer. Set Content-Type via binary field mimeType so n8n passes it correctly to Drive API.
3 emails sent per document instead of 2 MED Intake notification and filing attorney nodes were active by default, firing on every run even when no attorney email was configured. Disabled Send Intake Notification and Send to Filing Attorney nodes. Only reviewer notification + completion email remain active.
Completion email showed N/A for reviewer name, decision, notes MED After Move Source to Archive, the node output is Drive metadata. The pipeline data (reviewer info, cert ID, doc URL) was lost. Added "Restore Pipeline" Code node between Move to Archive and Send Completion Notification. Pulls data back from Merge Doc Result using $() reference.
Workflow crashed at Assemble Certificate. "Cannot read .slice of undefined" HIGH data.fileHash.slice(0,8) crashed when fileHash was undefined after Apply Review Decision couldn't recover pipeline state from the form data. Changed to (data.fileHash || 'NOCERT').slice(0,8) with null-safe fallback.
Google Drive trigger never detected new files after restart HIGH Deactivating/reactivating the workflow resets the trigger checkpoint to "now". Files already in the Inbox were invisible to the next poll. Manually set staticData['node:New File in Inbox'].lastTimeChecked to a time before the upload via the n8n API.
$helpers.httpRequestWithAuthentication not available in Code node HIGH n8n Code nodes run in an external task runner where $helpers is not defined. Discovered when attempting to call Drive API from inside a Code node. Replaced Code node API call with a dedicated HTTP Request node using predefined OAuth2 credential type, which handles token injection automatically.

Review SLA Monitor

A companion workflow that watches the human review queue and escalates if no one responds.

What it does

  • Runs every 4 hours on its own schedule
  • Calls the n8n internal API to find all paused executions
  • Filters for executions waiting at the Human Review node
  • Sends a reminder email after 4 hours with a direct form link
  • Sends an escalation email after 24 hours if still unresolved

Why it is a separate workflow

  • The main pipeline is fully suspended at the Wait node
  • Suspended executions cannot run any code or send emails
  • The SLA Monitor runs independently on its own trigger
  • It finds paused executions via API without interfering with them
  • Without it, a stuck document would sit silently with no alert
Nodes: Schedule Trigger → Fetch Waiting Executions → Filter Stale Reviews → Is Escalation? → Send Escalation Email / Send Reminder Email

Business Analysis: Simplicity vs. Privacy

The 1-page analysis: why the tokenization architecture was the right call for a legal translation firm.

Simple Architecture (no tokenization)

  • Send raw scanned PDF directly to GPT-4o
  • Faster to build. 2-3 nodes nodes instead of 8
  • No mapping/restore logic
  • Client PII stored on OpenAI servers
  • Risk: GDPR / attorney-client privilege violations
  • Not suitable for court-certified translations

Privacy-Preserving Architecture (this build)

  • PII tokenized before any AI call
  • AI model never sees real client identity
  • Token map lives only in execution memory
  • Glossary grows with every document
  • Human review ensures legal accountability
  • Scales to any language pair. Swap the prompt
Bottom line: For a firm serving immigration courts and law firms, the extra complexity of tokenization is not overhead — it is the product. The architecture makes the automation legally defensible and client-trustworthy in a way that a raw-send approach never can be.
